## 项目概述

这是机器人方向的第一个子工程。整体机器人项目会不断增加新的实现和设计记录；第一版选择 **语音小狗**，目标是让机器狗听懂语音命令，并通过姿态、头部动作、声音反馈和动作序列表达不同状态。

第一版不先重做主控板，而是使用 Petoi Bittle X V2 和 BiBoard V1，把工程重点放在可展示、可复现的交互效果上：语音输入、动作执行、表情化动作设计、调试记录和资料整理。

这个项目在网站里不是一个孤立工具，而是一个完整的作品集工程：主站项目页负责讲清楚设计思路和工程价值，`/robotics/voice-dog/` 负责保存可执行的构建清单、资料索引和迭代进度。

## 核心问题

我希望解决的问题是：一个入门级四足机器人怎样从“能动”变成“能交互”？如果只是让机器狗走路，它更像一个机械练习；如果它能听懂命令，并用可识别的身体语言回应，它就开始接近一个有交互性的机器人原型。

因此第一版聚焦三个问题：

1. **语音输入是否稳定：** 在普通室内环境下，机器狗能否可靠识别常用命令。
2. **动作是否可复现：** 坐下、站立、行走、问候、展示序列能否连续运行。
3. **表情是否可理解：** 不依赖屏幕，也能用姿态、头部动作和声音表达开心、倾听、疑惑、兴奋、困了。

## 我做了什么

1. 建立机器人分类页面，把机器人相关工程统一放到 `/robotics/` 下。
2. 创建第一个子工程 `/robotics/voice-dog/`，作为语音小狗的构建指南。
3. 整理零件清单、组装流程、固件上传、舵机校准、语音命令测试和自定义动作。
4. 设计第一版“表情动作”：开心、倾听、疑惑、兴奋、困了。
5. 保留旧 `/smartdog/` 路径作为兼容跳转，后续维护集中到机器人分类下。

## 表情与动作设计

第一版的“表情”不做屏幕表情，而是用身体语言实现：

| 状态 | 语音触发 | 行为设计 |
| --- | --- | --- |
| 开心 | Happy | 快速点头，小步向前，短音效反馈，然后回到站立 |
| 倾听 | Listen | 左右歪头，保持静止，像是在等待下一条命令 |
| 疑惑 | Confused | 小步后退，摇头，然后坐下 |
| 兴奋 | Dance | 执行 moonwalk 或 Skill Composer 短动作序列 |
| 困了 | Sleep | 坐下或趴下，进入安静状态 |

这种设计的好处是：即使没有 OLED/LED 面部模块，也能先验证“声音 -> 机器人状态 -> 可见反馈”的交互链路。之后再加入屏幕表情或视觉识别时，已有动作系统可以继续复用。

## 技术路线

- **硬件：** Petoi Bittle X V2、BiBoard V1、P1S 舵机、电池、可选视觉/超声波模块。
- **固件：** OpenCatESP32。
- **交互：** Petoi Voice Command Module。
- **动作设计：** 内置动作 + Skill Composer 自定义序列。
- **展示：** 静态网页构建清单、进度勾选、资料索引和验收标准。

## 站点内容设计

为了让这个工程不只是一个可运行系统，我把内容拆成三层：

1. **主页亮点：** 在首页作为 Research & Leadership Highlights 的一部分展示。
2. **作品集项目页：** 说明为什么做、怎么设计、体现哪些工程能力。
3. **构建指南页：** 在 `/robotics/voice-dog/` 中保存具体步骤、清单、风险排查和资料链接。

这种结构适合长期更新：当后续加入视觉避障、OLED 表情或 Raspberry Pi 控制时，可以继续在机器人分类中增加新版本，而不用重写整个站点。

## 第一版验收标准

- 机器狗能稳定开机。
- 舵机校准后能站立、坐下和行走。
- 至少 5 条内置语音命令可用。
- 至少 3 条自定义语音命令可用。
- 至少 3 个表情动作可以稳定展示。
- 完整演示序列可以连续运行两次，不需要手动重置。

## 下一步

下一阶段可以加入 LED/OLED 面部表情、视觉识别、避障行为或 Raspberry Pi 高层控制。当前版本先保证语音触发和动作表达稳定，再进入更复杂的感知与自主行为。

## 项目意义

这个项目体现的是从资料调研到工程实现的完整过程：选型、约束分析、硬件构建、固件配置、交互设计、测试验收和公开文档。它不是单纯“买一个机器人套件”，而是把现成硬件转化成一个可解释、可复现、可继续迭代的机器人交互项目。
